# Interview Guide: Math Problems

This document provides insights into common pitfalls, edge cases, variations, and interview tips specific to the math problems covered in this project.

## General Interview Tips for Math Problems

1.  **Clarify the Problem:** Always ask clarifying questions.
    *   What are the constraints on input numbers (positive, negative, zero, range, integer/float, `long long` vs `int`)?
    *   What are the expected outputs for edge cases (e.g., `GCD(0,0)`, `x^0`, `Fib(-1)`)?
    *   What is the desired time and space complexity?
2.  **Start with Brute-Force (if applicable):** Even if not optimal, explaining a brute-force approach first demonstrates understanding and provides a baseline for optimization.
3.  **Think about Edge Cases Early:** Zero, one, negative numbers, maximum/minimum integer values, floating-point precision issues.
4.  **Consider Overflows:** Especially with multiplication (e.g., LCM, factorials, large Fibonacci numbers), `long long` in C++ is often necessary. Be explicit about this.
5.  **Test Cases:** Propose diverse test cases including base cases, typical cases, and edge cases.
6.  **Explain Your Thought Process:** Talk through your logic, why you chose a particular algorithm, and how you're optimizing.
7.  **Time and Space Complexity:** Be ready to analyze and justify the complexity of your solution.
8.  **Alternative Approaches:** Be prepared to discuss other ways to solve the problem, even if you don't implement them.

## 1. Greatest Common Divisor (GCD) & Least Common Multiple (LCM)

### Edge Cases and Gotchas:
*   **Zero Inputs:**
    *   `GCD(A, 0) = |A|`. The Euclidean algorithm naturally handles this: `GCD(A, 0)` returns `A`.
    *   `GCD(0, 0)`: Conventionally 0, or sometimes considered undefined. The implemented `gcd_naive` returns 0; `gcd_euclidean` also returns 0 after `abs(0)=0`.
    *   `LCM(A, 0) = 0`. The formula `(|A*B|) / GCD(A,B)` works correctly here if GCD(A,0)=A.
*   **Negative Inputs:**
    *   GCD is usually defined for positive integers. `GCD(A, B) = GCD(|A|, |B|)`. All implementations correctly take absolute values.
*   **Large Numbers:** For `long long` inputs, the Euclidean algorithm is efficient. Naive GCD will be too slow.
*   **LCM Overflow:** `(A * B) / GCD(A, B)` can overflow `long long` if `A * B` is too large, even if the final LCM fits. Calculate as `(A / GCD(A, B)) * B` to mitigate this.

### Interview Tips and Variations:
*   **Why Euclidean is better:** Explain its logarithmic time complexity vs. linear for naive.
*   **Relationship between GCD and LCM:** Clearly state `LCM(A, B) = (|A * B|) / GCD(A, B)`.
*   **Extended Euclidean Algorithm:** For `ax + by = GCD(a, b)`. A common follow-up where you need to find `x` and `y`.
*   **GCD of multiple numbers:** `GCD(a, b, c) = GCD(GCD(a, b), c)`.
*   **LCM of multiple numbers:** `LCM(a, b, c) = LCM(LCM(a, b), c)`.
*   **Using `std::gcd` (C++17):** Be aware of it, but implement your own if asked.

## 2. Prime Number Generation (Sieve of Eratosthenes)

### Edge Cases and Gotchas:
*   **`limit < 2`:** Primes are defined for numbers `>= 2`. Handle `limit = 0` or `limit = 1` appropriately (empty list or `is_prime[0]=F, is_prime[1]=F`).
*   **Space Usage:** The boolean array requires `O(N)` space. For very large `N` (e.g., 10^9), this might be too much memory. Explain this limitation.
*   **Data Types:** `limit` can be `int`, but ensure intermediate calculations (like `p * p`) don't overflow if `limit` is close to `INT_MAX`.

### Interview Tips and Variations:
*   **When to use Sieve vs. `is_prime_naive`:** Sieve is for finding *all* primes up to N. Naive is for checking *one* number.
*   **Optimizations for Sieve:**
    *   Start marking multiples from `p*p` (already implemented).
    *   Only consider odd numbers after 2 (reduces memory and iterations by half, but slightly more complex index management).
    *   Segmented Sieve: For very large limits where the boolean array doesn't fit in memory. You sieve segments of numbers.
*   **Prime Factorization:** A common follow-up. You can use the `is_prime` array generated by Sieve to efficiently find prime factors.
*   **Number of primes up to N (`pi(N)`):** Roughly `N / log N`. Useful for estimating list size.

## 3. Power (x^n) Calculation

### Edge Cases and Gotchas:
*   **`n = 0`:** `x^0 = 1`. This is a crucial base case.
*   **`x = 0`:**
    *   `0^positive = 0`.
    *   `0^0 = 1` (mathematical convention, but sometimes undefined in specific contexts; clarify with interviewer).
    *   `0^negative` is undefined (division by zero). Implementations usually return 0.0 or `inf`/`nan`. Returning 0.0 is a pragmatic choice for competitive programming/interviews unless specified.
*   **`n < 0`:** `x^(-n) = 1 / x^n`.
*   **`x = 1` or `x = -1`:**
    *   `1^n = 1` for any `n`.
    *   `(-1)^n = 1` if `n` is even, `-1` if `n` is odd.
*   **Floating-point precision:** `double` has limited precision. Large exponents can lead to significant errors. State this limitation. If exact integer results are needed, a specialized large number library might be required.
*   **`INT_MIN` as exponent:** `abs(INT_MIN)` is still `INT_MIN` in two's complement. Use `long long` for `exp_abs` to handle this.

### Interview Tips and Variations:
*   **Binary Exponentiation:** Crucial to know this for `O(log n)` solution. Explain how it works by observing the binary representation of `n`.
*   **Integer Power:** If `x` is an integer and `n` is large, can use modular exponentiation (`(a^b) % m`) to avoid overflow and work with very large numbers.
*   **Library functions:** Be aware of `std::pow()` but explain you'd implement it yourself if asked.
*   **Recursive vs. Iterative Binary Exponentiation:** Discuss trade-offs (stack space vs. iterative control flow). Both are `O(log n)` time.

## 4. Nth Fibonacci Number

### Edge Cases and Gotchas:
*   **`n < 0`:** Fibonacci is typically defined for `n >= 0`. Return -1 or throw an error.
*   **`n = 0`, `n = 1`:** Base cases `F(0)=0`, `F(1)=1`.
*   **Overflow:** Fibonacci numbers grow exponentially. `F(93)` is the largest that fits into a `long long`. For larger `N`, you need arbitrary-precision arithmetic (BigInt library). State this limitation.
*   **Performance of Naive Recursion:** Emphasize the `O(2^n)` complexity and why it's unacceptable for `n > ~40`.

### Interview Tips and Variations:
*   **Comparison of Approaches:** Clearly articulate the time and space complexity differences between naive recursive, memoized recursive, and iterative solutions. Iterative is generally preferred for `O(1)` space.
*   **Dynamic Programming:** Explain how memoization (top-down DP) and iterative (bottom-up DP) solve the problem.
*   **Matrix Exponentiation:** For very large `N` (e.g., up to 10^18), matrix exponentiation can compute `F(N)` in `O(log N)` time. This is an advanced technique and a great follow-up question.
    *   `[[F(n+1)], [F(n)]] = [[1, 1], [1, 0]]^n * [[F(1)], [F(0)]]`
*   **Closed-form expression (Binet's Formula):** `F(n) = (Phi^n - Psi^n) / sqrt(5)`. While mathematically elegant, it involves floating-point numbers which can lead to precision issues for large `n`. Not typically used for exact integer computation in interviews unless specifically for properties or approximations.
*   **Variations:**
    *   `F(n) = F(n-1) + F(n-2) + F(n-3)` (Tribonacci sequence) - DP solutions generalize easily.
    *   Finding the Nth Fibonacci number modulo M. The iterative DP works, but there's also a Pisano period optimization for very large N.
    *   Finding if a number is a Fibonacci number (check if `5*n^2 + 4` or `5*n^2 - 4` is a perfect square).
